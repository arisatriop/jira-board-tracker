# Done — Post Jira Ticket Comment

Post a work summary comment to a Jira ticket when you finish working on it.

Usage: `/done TICKET_ID` (e.g. `/done PROJ-123`)

1. Read credentials from `config/.env` using grep:
   ```bash
   grep -E "^JIRA_URL=" config/.env | cut -d= -f2-
   grep -E "^JIRA_EMAIL=" config/.env | cut -d= -f2-
   grep -E "^JIRA_API_TOKEN=" config/.env | cut -d= -f2-
   ```
   If any of the three are missing or empty, stop and tell the user which vars to add to `config/.env`.

2. Run these in parallel to gather context:
   - `git log main...HEAD --oneline` — commits on this branch
   - `git diff main...HEAD --stat` — files changed summary
   - `git branch --show-current` — current branch name

3. Use the GitHub MCP tool to find the open PR for the current branch. Get its URL if found.

4. Based on the commits and changed files, infer QA test steps: what endpoints or features changed, what inputs/scenarios QA should verify, and what the expected outcome is. Be specific — reference actual route paths, request bodies, or UI flows if derivable from the diff. Aim for 3–5 actionable steps.

5. Compose the comment text:
   ```
   ✅ Work completed on [branch-name] — <one-sentence summary of what was done, inferred from commits and changed files>

   *Detail*
   <2–4 sentences explaining what changed and why — what problem it solves, what was added/modified/removed, and any important implementation notes. Written in plain language for a non-technical reader.>

   *Commits*
   - <each commit from step 2, one per line>

   *Files changed*
   <paste the git diff --stat output>

   *Pull Request*
   <PR URL, or "No open PR found">

   *How to test*
   1. <QA step 1>
   2. <QA step 2>
   3. <QA step 3>
   ...

   ---
   Generated by [Claude|https://claude.ai/code]
   ```

6. Show the composed comment to the user and ask: "Post this comment to [TICKET_ID]? (yes/no)". Wait for confirmation. If the user says no or requests edits, apply the changes and show the updated comment before asking again. Only proceed once confirmed.

7. Write the comment to a temp file as JSON, then post it with curl:
   ```bash
   cat > /tmp/jira_comment.json << 'EOF'
   {"body": "<comment text with \n for newlines, special chars escaped>"}
   EOF

   curl -s -o /tmp/jira_response.json -w "%{http_code}" \
     -X POST "$JIRA_URL/rest/api/2/issue/$TICKET_ID/comment" \
     -u "$JIRA_EMAIL:$JIRA_API_TOKEN" \
     -H "Content-Type: application/json" \
     -d @/tmp/jira_comment.json
   ```

8. Check the HTTP status code:
   - `201` — success. Confirm to the user: "Comment posted to $TICKET_ID."
   - `401` — credentials invalid. Tell the user to check `JIRA_EMAIL` and `JIRA_API_TOKEN` in `config/.env`.
   - `404` — ticket not found. Tell the user to verify the ticket ID.
   - Other — show the response body from `/tmp/jira_response.json` for debugging.

9. Clean up: `rm -f /tmp/jira_comment.json /tmp/jira_response.json`
